Chapter 2 — The Problem

Password management is broken

  • 💻
    Software password managers live on your device — and vanish or become inaccessible when that device does.
  • ☁️
    Cloud sync means your secrets are someone else's problem too. Breach one provider, expose millions.
  • 🧠
    Memorising passwords leads to reuse, weak patterns, and the sticky note under the keyboard.
  • 📱
    Cross-device consistency is painful — one manager for your work laptop, another on your personal machine, something else on your phone and tablet. They never quite sync, and each platform has its own quirks.
  • 🔑
    Hardware tokens exist — but they're expensive, proprietary, or too limited.
Chapter 2 — Attack Vectors

Cloud vs Offline — where can you be hit?

☁️ Cloud Password Manager

  • 🏢
    Provider breach — your vault travels with the attacker. (LastPass 2022, others)
  • 🎣
    Phishing & account takeover — one fake login page unlocks everything.
  • 🧩
    Malicious browser extension — sits in the same process as the vault UI.
  • 📋
    Clipboard hijacking — malware reads the password the moment it's copied.
  • 🌐
    Sync endpoint MITM — traffic between devices can be intercepted.
  • 👤
    Insider threat — provider employees with elevated access.

🔒 ESPWV32 (Offline Hardware)

  • 🔌
    No provider, no breach — there is no server to compromise.
  • 📵
    No phishing surface — credentials never leave the device as text.
  • ⌨️
    Appears as a keyboard — the host OS sees keystrokes, not a secret store.
  • 📦
    Physical theft — mitigated by PIN + AES-256. Data unreadable without both.
  • 📡
    BLE eavesdropping — blocked by SC+MITM bonding with a device-side passkey.
  • 💾
    Raw EEPROM dump — AES-256-CBC encryption renders it unreadable.

Chapter 5 — Security

Security by design

🔐

BLE Secure Pairing (MITM-protected)

Uses SC+MITM bonding with a random passkey displayed on the device — the host never chooses the PIN.

🔑

AES-256-CBC Credential Encryption

Every credential slot is encrypted at rest. Nothing is readable without the correct PIN — even with direct EEPROM access.

🧂

PBKDF2-HMAC-SHA256 PIN Verification

The PIN is never stored. A salted, 10,000-iteration key derivation function verifies it — brute force is computationally costly.

📵

Zero Cloud Surface

No sync, no server, no account. Data lives only on your device. The WiFi interface is local-only and AP-isolated.

Chapter 5 — Lessons Learned

It wasn't easy

📚

The BLE spec is enormous

The Bluetooth core spec runs to thousands of pages. On top of that: GAP, GATT, SMP, and the HID-over-GATT profile — each its own document.

Finding the exact combination of ESP_LE_AUTH_REQ_* and ESP_IO_CAP_* flags that actually worked took many failed pairing attempts.

The ESP32 Arduino BLE library also had breaking API changes between major versions — code that worked one week silently broke the next.

🧭

Started before AI

This project began before tools like ChatGPT or Copilot were available. Progress was pieced together from:

  • 📄 Bluetooth spec PDFs
  • 🔍 Stack Overflow fragments
  • 🐛 ESP32 Arduino GitHub issues
  • 💬 Arduino forum posts
  • 📦 T-vK's ESP32-BLE-Keyboard library as a starting point

Each session was a slow excavation of tribal knowledge.

💀

Debugging in the dark

When the device runs out of heap — for instance from BLE and WiFi running simultaneously without releasing Classic BT memory — it doesn't throw an error. It just browns out and resets silently.

The only window into the device is the serial monitor at 115200 baud over USB. No stack traces. No breakpoints. Serial.println() is your debugger.

Fixes like calling esp_bt_controller_mem_release() before BLEDevice::init() were found only after hours of brown-out loops with no obvious cause.

"The device doesn't tell you what's wrong — it just stops."

Chapter 5 — Bluetooth Deep Dive

BLE pairing & MITM protection

LE Secure Connections (SC)

ESPWV32 uses Bluetooth LE Secure Connections with SC_MITM_BOND — the strongest BLE pairing mode. The key agreement runs over ECDH P-256, meaning the session key is mathematically unrecoverable even if all BLE traffic is recorded.

Why MITM is blocked

With ESP_IO_CAP_OUT (display-only), ESPWV32 shows a random passkey on its physical screen. An attacker who intercepts the BLE packets still needs to read that screen — which requires being physically present at the device. Remote interception becomes impossible.

Bonding

After the first pairing the LTK is stored on both sides. Future connections re-use the stored key — no passkey re-entry — but the encryption is identical. Unpair to revoke.

ECDH P-256 ESP_IO_CAP_OUT LTK bonding MITM-proof
Chapter 5 — HID Deep Dive

How the HID report descriptor works

Report 1 — Keyboard (8 bytes)

Report 2 — Media Keys (2 bytes / 16 bits)

What is a report descriptor?

A HID report descriptor is a byte array sent to the host at connection time. It tells the OS the exact binary layout of every packet the device will ever send — field names, sizes, value ranges, and usage IDs. The OS uses it to build the driver on the fly. No custom driver is ever needed.

Report 1 — Keyboard

8 bytes per report. Byte 0 carries 8 modifier-key bits (Shift, Ctrl, Alt, Meta — left and right variants). Bytes 2–7 each hold one USB HID keycode, allowing up to 6 simultaneous keypresses (6KRO). ESPWV32 only ever fills byte 0 + bytes[0] — one key at a time — exactly like a real keyboard.

Report 2 — Media / Consumer

16 individual bits, each mapped to a consumer control action (volume, playback, browser navigation…). Included for completeness — ESPWV32 does not currently use this report.

Chapter 6 — Polish

The details that matter

🌙

Screen dimming

Automatically dims after 15 s idle, turns off at 30 s. Any button press restores full brightness instantly. Disabled in admin modes.

🔀

WiFi password randomisation

The admin hotspot password is auto-generated on first boot and can be regenerated from the web UI — no static, guessable password.

🔋

Battery indicator

Live percentage shown on every screen, updated every 30 s. Turns red below 30 %. Shows a lightning bolt while charging.

📡

BLE / WiFi coexistence

Classic Bluetooth memory is released at boot to free heap. BLE gets radio priority at startup; WiFi coexistence is negotiated when the admin AP starts.

💾

Adaptive EEPROM sizing

Probes the largest flash page the ESP32 library supports, halving until it succeeds — maximising credential slots on any board variant.

🔁

Credential send modes

Toggle between sending username+password, username only, or password only with a medium button press — without disconnecting.

Reference

Glossary

Security & Cryptography
AESAdvanced Encryption Standard — symmetric block cipher used to encrypt credential slots at rest.
AES-256-CBCAES with a 256-bit key in Cipher Block Chaining mode — each block's ciphertext depends on the previous.
CBCCipher Block Chaining — AES mode where each plaintext block is XOR'd with the previous ciphertext block before encryption.
ECDHElliptic Curve Diffie-Hellman — key agreement protocol used during BLE LE Secure Connections pairing.
HMACHash-based Message Authentication Code — used inside PBKDF2 to stretch and verify the PIN.
IVInitialization Vector — a non-secret random value that ensures identical plaintexts produce different ciphertexts in CBC mode.
LTKLong Term Key — the shared secret stored after a successful BLE pairing, used to re-encrypt future connections without re-pairing.
MITMMan-In-The-Middle — an attack where a third party secretly intercepts communication between two parties.
P-256NIST P-256 (secp256r1) — the elliptic curve used in BLE LE Secure Connections ECDH key exchange.
PBKDF2Password-Based Key Derivation Function 2 — applies HMAC-SHA256 thousands of times to make PIN brute-force computationally expensive.
SCSecure Connections — BLE LE pairing mode based on ECDH P-256 that provides forward secrecy and MITM protection.
SHA-256Secure Hash Algorithm (256-bit digest) — used inside PBKDF2-HMAC-SHA256 to hash the PIN verifier.
Bluetooth & HID
BLEBluetooth Low Energy — low-power wireless protocol used by ESPWV32 to act as a keyboard.
HIDHuman Interface Device — USB/BLE class for keyboards, mice, and gamepads. No custom driver required.
6KRO6-Key Rollover — the keyboard report supports up to 6 simultaneous keycodes in bytes 2–7.
LELow Energy — the "LE" in BLE; refers to the Bluetooth 4.0+ subset optimised for battery-powered devices.
Hardware & Networking
APAccess Point — the WiFi hotspot ESPWV32 creates (SSID: ESPWV32-Admin) for the web admin interface.
DNSDomain Name System — ESPWV32 runs a DNS server on the AP to redirect all hostnames to the captive portal.
EEPROMElectrically Erasable Programmable Read-Only Memory — flash-backed storage on the ESP32 used to persist encrypted credentials.
ESP32Espressif Systems 32 — dual-core Xtensa LX6 SoC with integrated WiFi and Bluetooth, the heart of the M5StickC.
SoCSystem on a Chip — a single integrated circuit containing CPU, RAM, flash, WiFi, and Bluetooth (e.g. the ESP32).
SSIDService Set Identifier — the human-readable name of a WiFi network; ESPWV32 broadcasts "ESPWV32-Admin".
USBUniversal Serial Bus — used for the serial admin connection (115200 baud) and device charging.
8N18 data bits, No parity, 1 stop bit — the serial framing used by the ESPWV32 serial admin interface.
Other
COMComponent Object Model — Windows API used by the Icecriminator to script Outlook via PowerShell.
CRUDCreate, Read, Update, Delete — the four basic data operations supported by the WiFi admin web interface.
PINPersonal Identification Number — the 4-digit code used to unlock ESPWV32 and decrypt the credential store.
TOTPTime-based One-Time Password — a planned feature to generate 2FA codes directly on the device (RFC 6238).
WebAuthnWeb Authentication API — a phishing-resistant standard (FIDO2) that eliminates passwords via hardware-bound keys.
